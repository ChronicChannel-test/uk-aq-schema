# UK AQ v0.2.0 schema migration notes

## Canonical target

The files under `schemas/v0.2.0/` describe the final v0.2.0 target for a clean database. In that target:

- observations has no connector_id and uses primary key (timeseries_id, observed_at)
- timeseries links directly to observed_properties and has no phenomenon_id
- stations belongs to one connector and one network
- stations.match_id links duplicate source stations through station_matches
- station_matches.match_key is an optional generated matching key
- station_initial_metadata is an insert-once snapshot

The canonical files are not the migration procedure for an existing database.

## Phase 1 default migrations

The default TEST migration sequence is additive:

1. `001_preflight_checks.sql`
2. `002_add_v020_core_tables_and_columns.sql`
3. `003_seed_networks_and_connectors.sql`
4. `004_migrate_station_fields_and_initial_metadata.sql`
5. `005a_migrate_timeseries_observed_properties.sql`
6. `005b_validate_timeseries_observed_properties.sql`
7. `006_migrate_ingest_runs.sql`
8. `007_migrate_placeholder_rows.sql`
9. `008_rebuild_public_views_and_rpcs.sql`
10. `009_validation_checks.sql`
11. `010_dependency_report_sql_checks.sql`

Phase 1 adds networks, station matches, initial metadata, station network/match fields, observed-property links, ingest-run network context, and observation status/metadata where missing.

Phase 1 deliberately retains:

- observations.connector_id, partitioning and legacy primary key
- timeseries.phenomenon_id and other legacy bridge columns
- station_metadata
- phenomena
- station_network_memberships
- uk_aq_networks
- existing public views, RPCs, AQI functions and ingest contracts

`005b` and `010` are read-only reports. Review and save their output.

## Breathe London matching

`device_code` or `DeviceCode` may populate stations.station_device_ref. The complete metadata object, including InstallationCode, is copied once to station_initial_metadata.

InstallationCode is not a device code. A later matching phase can join a Nodes InstallationCode to a Communities station_ref and use:

```text
station_matches.match_key = blondon_installation:<InstallationCode>
```

## Deferred enforcement and hard cut

Files numbered `900` and above are not part of phase 1:

- `900_hard_cut_observations_after_dependencies.sql`
- `901_validate_hard_cut_observations.sql`
- `905_enforce_timeseries_observed_property_not_null.sql`

The observation hard cut is deliberately disabled with a blocking exception until its approved security section has been implemented. It must not run until:

- all view, RPC, AQI and ingest dependencies use the v0.2.0 contracts
- observation writes are paused in a maintenance window
- null timeseries IDs, orphan rows and new-key collisions are zero
- source and target row-count checks are approved
- ownership, RLS, policies, grants, comments, indexes and triggers are recreated
- rollback and recovery have been tested

The hard cut must run in an explicit transaction and lock observations before copying. It must never use `DISTINCT ON` or filtering to conceal rows that cannot fit the new primary key.

`905` must remain deferred until `005b` reports zero unmapped timeseries and every writer supplies observed_property_id.

## Dependency work

The database-side report is `010_dependency_report_sql_checks.sql`. The source-code inventory is in `docs/v0.2.0_dependency_check.md`.

Required code work includes:

- replace observation writes and conflict keys
- replace phenomena joins with observed_properties joins
- replace mutable station_metadata writes with dedicated station columns or other purpose-specific storage
- replace membership tables with stations.network_id and networks
- rebuild public projections and AQI functions
- update snapshot, history, dashboard and website consumers

No SQL in this bundle has been applied automatically.
